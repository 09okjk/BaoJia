# 智能报价反馈飞轮设计方案

## 1. 文档目标

本文档用于为当前智能报价系统设计一套可落地的“用户反馈 -> 结构化记忆 -> 相似检索 -> 下次生成更合理报价”的反馈飞轮方案。

方案目标不是立即做不可控的模型自学习，而是在现有 Skill + Workflow 架构基础上，先实现：

1. 结构化采集用户对报价草案的反馈
2. 将反馈沉淀为可检索、可追溯、可审核的记忆
3. 在后续相似报价中注入这些记忆，减少重复错误
4. 对高频、稳定、低争议的反馈形成规则候选
5. 在不破坏当前 QuoteDocument 契约的前提下，逐步提升首稿质量

## 当前实现状态

截至当前版本，反馈飞轮 MVP 已在仓库内落地，当前已实现：

1. `quote_feedback_capture_skill`
2. `quote_feedback_reference_skill`
3. `quote_feedback_rule_review_skill`
4. `.opencode/memory/` 本地记忆目录
5. feedback memory 持久化写入器
6. memory 去重与 observation 升级
7. rule candidate 审核与 approved rules 写入
8. `quote_orchestration_skill` 接入 `feedback_reference`
9. `quote_pricing_skill` 与 `quote_review_output_skill` 的最小消费逻辑

当前尚未实现：

1. 前端审核列表视图
2. approved rules 停用 / 回滚管理
3. 更细粒度的 remark / payment term 精准消费
4. 将 approved rules 同步为显式静态 pricing 配置

## 2. 设计原则

### 2.1 先做可控记忆，不直接做黑盒学习

报价属于高风险业务。

系统不应在缺少审核和归因的情况下，直接把用户自由文本反馈写入模型参数或直接改变全部生成逻辑。

本方案优先采用：

1. 结构化反馈采集
2. 相似反馈检索增强
3. 规则候选归纳
4. 经人工审核后固化稳定规则

### 2.2 反馈必须绑定到具体对象

系统不能只保存“这份报价不太好”之类模糊评价。

反馈必须尽量绑定到具体目标，例如：

1. `header` 字段
2. `option`
3. `section`
4. `group`
5. `line`
6. `footer.remark`
7. `service_payment_terms`
8. `review_result`

### 2.3 学习结果必须分层

不同类型的反馈，不应使用同一套生效方式。

建议分为四层：

1. `hard_rule`：明确约束，命中即执行
2. `soft_preference`：偏好信息，影响排序或文案选择
3. `negative_pattern`：负样本，降低类似内容被自动注入的概率
4. `review_alert`：仅用于审核提醒，不自动改动内容

### 2.4 反馈飞轮必须可追溯

任何一次自动调整，都需要能回答：

1. 是基于哪条反馈记忆触发的
2. 命中了哪个客户 / 船型 / 服务模式 / 设备上下文
3. 是硬规则、软偏好还是负样本
4. 是否经过人工审核

## 3. 目标问题

当前系统虽然已经能稳定生成 `QuoteDocument`，但首稿仍可能出现以下问题：

1. 自动补入了业务上不适用的收费项
2. 历史 remark 与当前项目不匹配
3. 某些客户或业务员有稳定偏好，但系统无法继承
4. 用户修改过同类问题后，下次仍重复出现
5. 相似项目的最终成稿经验没有反哺到前置生成阶段

反馈飞轮要解决的是：

1. 让系统学会“哪里常错”
2. 让系统学会“类似场景下应该怎么改”
3. 让系统逐步从“通用草案”进化为“更贴合客户与业务习惯的草案”


## 5. 飞轮闭环

### 5.1 第一步：生成草案

系统按照当前 workflow 生成：

1. `quote_request`
2. `feasibility_result`
3. `historical_reference`
4. `pricing_result`
5. `quote_document`

并保留中间 trace：

1. `line.basis`
2. `trace.historical_references`
3. `trace.pricing_basis`
4. `trace.rule_versions`

### 5.2 第二步：用户反馈

用户不直接“修改 PDF”，而是对草案中的结构化对象给出反馈：

1. 删除某一行
2. 把某一行改成 `pending`
3. 修改 remark 文案
4. 调整 payment terms
5. 修改某个表头字段
6. 指出某类历史补项不应出现

### 5.3 第三步：反馈结构化

系统将用户反馈转成标准 `feedback_event`。

一个 `feedback_event` 必须至少表达：

1. 反馈作用对象是谁
2. 用户做了什么动作
3. 原因是什么
4. 希望以后在什么范围内复用
5. 是否已经被人工确认

### 5.4 第四步：记忆沉淀

反馈进入记忆层后，不直接修改全部规则，而是先写入：

1. 反馈事件表
2. 案例记忆表
3. 偏好记忆表
4. 规则候选表

当前实现中，以上内容已写入 `.opencode/memory/`：

1. `.opencode/memory/feedback-events/`
2. `.opencode/memory/case-memory/`
3. `.opencode/memory/preference-memory/`
4. `.opencode/memory/rule-candidates/`
5. `.opencode/memory/approved-rules/`

### 5.5 第五步：下次生成前检索

下次生成类似报价时，系统先检索：

1. 相似反馈事件
2. 当前客户偏好
3. 当前业务员偏好
4. 当前服务模式 / 船型 / 设备模式的稳定反例

并将结果注入到后续 Skill 中。

### 5.6 第六步：稳定反馈升级为规则

当同类反馈满足如下条件时，可以升级为规则候选：

1. 重复出现 >= 3 次
2. 作用对象一致
3. 适用上下文稳定
4. 修改方向一致
5. 无明显业务冲突

规则候选经人工审核后，才能进入正式规则层。

当前实现中的升级路径为：

1. feedback 重复出现后累积 `observation_count`
2. preference 在 `observation_count >= 2` 时升级为 `active`
3. rule candidate 在 `observation_count >= 3` 时升级为 `ready_for_review`
4. `quote_feedback_rule_review_skill` 人工审核后写入 `approved-rules`

## 6. 记忆分层模型

建议分四类记忆。

### 6.1 Case Memory

表示一次完整报价案例，包括：

1. 输入评估单与补充上下文
2. 初稿输出
3. 用户修改后的定稿
4. 修改 diff
5. 是否最终接受

作用：

1. 用于相似案例检索
2. 用于对比“初稿 -> 定稿”的改动模式

### 6.2 Feedback Memory

表示某一条明确反馈事件。

作用：

1. 精确记录“哪里错了”
2. 用于下次命中相似问题

### 6.3 Preference Memory

表示客户、业务员、区域、模板或设备类型的稳定偏好。

作用：

1. 在生成时提前改变候选排序
2. 在 review 阶段补充偏好提醒

### 6.4 Rule Candidate Memory

表示从多次反馈归纳出来、尚未正式生效的规则候选。

作用：

1. 供管理员审核
2. 审核通过后进入正式规则配置

### 6.5 Approved Rule Memory

表示已经完成人工审核、可以直接参与后续报价生成的正式规则。

作用：

1. 以高于 raw feedback 与普通 preference 的优先级参与 feedback reference
2. 为后续相似报价提供更稳定的抑制或调整依据

## 7. 核心数据模型

## 7.1 feedback_event

推荐结构：

```json
{
  "feedback_id": "fb-2026-0001",
  "quote_id": "WK-dimitra-m-2026-04-01",
  "quote_version": "draft-v3",
  "target": {
    "target_type": "line",
    "target_id": "option-1-line-hist-other-dockyard_management",
    "stage": "pricing"
  },
  "action": {
    "action_type": "remove",
    "before_value": {
      "status": "as_actual",
      "description": "Dockyard management fee"
    },
    "after_value": null
  },
  "reason": {
    "reason_code": "not_applicable",
    "reason_text": "This charge should not be included for this job."
  },
  "scope": {
    "scope_type": "customer",
    "scope_key": "customer:demo-customer"
  },
  "context_tags": {
    "customer_name": "Demo Customer",
    "vessel_type": "kamsarmax",
    "engine_model": "6S60ME-C8.2",
    "service_mode": "dock_repair",
    "location_type": "shipyard",
    "template_type": "engineering-service"
  },
  "accepted": true,
  "created_by": "sales_user_a",
  "created_at": "2026-04-15T10:00:00+08:00"
}
```

### 字段说明

1. `target.target_type`
   支持：`header_field`、`option`、`section`、`group`、`line`、`footer_remark`、`payment_term`、`review_flag`

2. `action.action_type`
   支持：`add`、`remove`、`replace`、`mark_pending`、`mark_owner_supply`、`change_text`、`change_amount`

3. `reason.reason_code`
   建议枚举：
   - `not_applicable`
   - `customer_preference`
   - `business_preference`
   - `historical_noise`
   - `pricing_too_high`
   - `pricing_too_low`
   - `scope_incomplete`
   - `should_be_pending`
   - `remark_not_acceptable`
   - `payment_term_not_acceptable`

4. `scope.scope_type`
   用于决定记忆复用范围：
   - `this_quote_only`
   - `customer`
   - `sales_owner`
   - `template_type`
   - `service_pattern`
   - `global_candidate`

## 8. 反馈采集设计

## 8.1 用户界面建议

用户不应面对一个纯文本输入框来表达所有修改意见。

建议前端提供以下操作：

1. 行级操作
   - 删除该行
   - 改为待确认
   - 改为船东提供
   - 改金额
   - 改说明

2. remark 级操作
   - 删除 remark
   - 修改文案
   - 标记为客户不接受

3. 表头级操作
   - 补客户名称
   - 补 attention
   - 修服务窗口

4. 生效范围选择
   - 仅本单生效
   - 对该客户生效
   - 对我本人后续报价生效
   - 提交为规则候选

## 8.2 反馈输入协议

建议新增一个统一反馈输入结构：

```json
{
  "quote_context": {
    "quote_id": "WK-dimitra-m-2026-04-01",
    "quote_document": {},
    "quote_request": {},
    "pricing_result": {}
  },
  "user_feedback": [
    {
      "target_type": "line",
      "target_id": "option-1-line-hist-other-dockyard_management",
      "action_type": "remove",
      "reason_text": "This line should not appear for this customer.",
      "scope_type": "customer"
    }
  ],
  "feedback_actor": {
    "user_id": "sales_user_a",
    "role": "sales"
  }
}
```

## 9. 新增 Skill 设计

当前已落地三个核心 Skill。

## 9.1 `quote_feedback_capture_skill`

### 目标

将用户对草案的修改意见转成结构化反馈事件，并输出可持久化的记忆对象。

### 输入

```json
{
  "quote_context": {},
  "user_feedback": [],
  "feedback_actor": {},
  "memory_options": {
    "persist": true,
    "memory_dir": ".opencode/memory"
  }
}
```

### 输出

```json
{
  "feedback_events": [],
  "case_memory_patch": {},
  "preference_candidates": [],
  "rule_candidates": [],
  "review_flags": [],
  "memory_write_result": {}
}
```

### 核心职责

1. 结构化用户反馈
2. 补齐 target 与上下文标签
3. 生成 `feedback_event`
4. 生成 `preference_candidate`
5. 生成 `rule_candidate`
6. 为持久化层提供标准输出

### 当前实现补充

当前 skill 已支持：

1. 直接写入 `.opencode/memory`
2. 返回 `memory_write_result`
3. 生成 `feedback_signature`
4. 对同类 feedback 进行聚合与 observation 计数

### 不负责

1. 直接修改正式规则配置
2. 直接微调模型

## 9.2 `quote_feedback_reference_skill`

### 目标

在生成报价前，检索与当前项目最相关的反馈记忆，并将其转换为可被 workflow 消费的建议。

### 输入

```json
{
  "quote_request": {},
  "feedback_context": {
    "customer_name": "",
    "sales_owner": "",
    "template_type": "engineering-service"
  }
}
```

### 输出

```json
{
  "matches": [],
  "applicable_preferences": [],
  "applicable_rules": [],
  "forbidden_patterns": [],
  "recommended_adjustments": [],
  "review_alerts": [],
  "confidence": 0.0
}
```

### 核心职责

1. 检索相似反馈事件
2. 提取客户偏好
3. 提取业务员偏好
4. 生成本次报价可用的抑制项和建议项
5. 输出信心分数与命中证据

### 当前实现补充

当前 skill 已支持读取：

1. `.opencode/memory/feedback-events/`
2. `.opencode/memory/preference-memory/`
3. `.opencode/memory/approved-rules/`

并会输出：

1. `applicable_rules`
2. approved rule 驱动的 `forbidden_patterns`
3. approved rule 驱动的 `recommended_adjustments`
4. approved rule 驱动的 `review_alerts`

## 9.3 `quote_feedback_rule_review_skill`

### 目标

审核已经达到稳定阈值的 rule candidate，并决定是否将其写入 approved rules。

### 输入

```json
{
  "rule_candidate_id": "rule-agg-customer-customer-abc-shipping-line_suppression-line-dockyard_management-remove",
  "decision": "approve",
  "review_note": "Repeatedly observed for the same customer. Approved as a stable suppression rule.",
  "memory_dir": ".opencode/memory",
  "reviewer": {
    "user_id": "manager_user_a",
    "role": "manager"
  }
}
```

### 输出

```json
{
  "rule_candidate": {},
  "approved_rule": {},
  "memory_write_result": {}
}
```

### 核心职责

1. 读取 `rule-candidates`
2. 执行 `approve / reject`
3. 审核通过后写入 `.opencode/memory/approved-rules`
4. 回写原 rule candidate 状态

## 10. Workflow 接入点

基于当前 `quote_orchestration_skill`，建议在 `prepare` 之后、`feasibility` 之前插入反馈检索节点。

推荐顺序：

```text
quote_template_select_skill（可选）
quote_request_prepare_skill
quote_feedback_reference_skill      <- 新增
quote_feasibility_check_skill
historical_quote_reference_skill（可选）
quote_pricing_skill
quote_review_output_skill
quote_pdf_render_skill（可选）
```

### 原因

1. `prepare` 之后，系统已获得标准化 `quote_request`
2. 这时上下文已足够稳定，适合做反馈检索
3. 后续的 feasibility / pricing / review 都能消费反馈记忆结果

当前实现已按上述顺序接入 `quote_orchestration_skill`，并在 workflow 输出中增加了 `feedback_reference` 中间态。

## 16. 用户体验建议

## 16.1 报价编辑页

建议页面结构为三栏：

1. 左侧：结构化编辑区
2. 中间：草案状态与审核提示
3. 右侧：HTML / PDF 预览区

页面底部建议固定提供两个明确动作按钮：

1. `继续修改`
2. `确认当前版本`

## 16.2 用户反馈动作

每个对象都支持：

1. `删除`
2. `改为待确认`
3. `修改文案`
4. `修改金额`
5. `标记为客户偏好`
6. `记住这次选择`

## 16.3 作用范围选择

用户在反馈时选择：

1. 仅本单
2. 记住为该客户偏好
3. 记住为我的个人偏好
4. 提交为规则候选

## 16.4 修改后动作

用户点击：

1. `保存反馈`
2. `重新生成草案`
3. `重新导出 PDF`

系统不是直接编辑 PDF，而是：

1. 更新结构化草案
2. 基于反馈重跑必要阶段
3. 再次渲染 PDF

## 16.5 用户显式决策机制

为了避免系统猜测“用户是否已经满意”，当前设计建议将纠错流程的结束条件交给用户显式选择，而不是由模型自动判断。

推荐只提供两个明确选项：

1. `继续修改`
2. `确认当前版本`

### `继续修改`

表示用户认为当前版本仍需调整。

系统行为：

1. 进入 `revising` 状态
2. 开放结构化修改入口
3. 记录本轮反馈并写入 memory
4. 基于反馈重跑必要阶段
5. 回到新的草案版本

### `确认当前版本`

表示用户接受当前草案，当前纠错流程结束。

系统行为：

1. 进入 `accepted` 状态
2. 将当前版本记录为本轮 accepted draft
3. 允许进入导出 PDF、提交审批等后续步骤
4. 不再继续本轮纠错循环

## 16.6 纠错流程状态机

建议最小状态机如下：

1. `drafting`
   - 系统正在生成草案
2. `awaiting_user_decision`
   - 草案已生成，等待用户在“继续修改 / 确认当前版本”之间二选一
3. `revising`
   - 用户选择继续修改，系统正在处理修改意见并重生成草案
4. `accepted`
   - 用户确认当前版本，当前纠错流程结束

推荐状态字段示例：

```json
{
  "draft_status": "awaiting_user_decision",
  "user_decision": null
}
```

当用户点击 `继续修改` 时：

```json
{
  "draft_status": "revising",
  "user_decision": "revise"
}
```

当用户点击 `确认当前版本` 时：

```json
{
  "draft_status": "accepted",
  "user_decision": "accept"
}
```

## 16.7 结束判断原则

当前设计明确规定：

1. 不以“用户沉默”作为满意信号
2. 不以“导出过 PDF”单独作为满意信号
3. 不以“修改次数减少”作为满意信号
4. 只有用户显式点击 `确认当前版本`，才视为当前纠错流程结束

### 当前实现状态

当前仓库已落地对话式决策能力：

1. `quote_orchestration_skill` 输出 `draft_status`
2. `quote_orchestration_skill` 输出 `user_decision`
3. 当草案生成完成且尚未确认时，输出 `user_decision_prompt`
4. 调用方可在下一轮对话中传入：
   - `user_decision: "accept"`
   - `user_decision: "revise"`
5. 其中：
   - `accept` 会将 `draft_status` 置为 `accepted`
   - `revise` 会将 `draft_status` 置为 `revising`

## 17. 风控要求

### 17.1 禁止无审核地自我覆盖规则

用户单次反馈只能进入记忆层，不能直接修改正式规则层。

### 17.2 强规则必须可回滚

任何升级为正式规则的候选，都必须支持：

1. 审核记录
2. 生效版本
3. 回滚

### 17.3 记忆要区分主体

必须区分：

1. 客户偏好
2. 业务员偏好
3. 团队习惯
4. 全局业务规则

否则系统会把局部偏好误学成全局规则。

### 17.4 不允许跳过不确定性表达

即使命中了历史反馈记忆，也不能把本来应 `pending` 的高风险事实伪装成确定报价。

5. 同一客户的重复错误率

## 21. 结论

当前仓库最合适的学习路线不是直接做模型微调，而是：

1. 把用户反馈采集成结构化事件
2. 把反馈沉淀为可检索记忆
3. 在下次报价前检索并注入 workflow
4. 将稳定高频反馈升级为规则候选
5. 用业务审核来控制学习质量

这样可以在保持当前 QuoteDocument 契约和 Skill 架构稳定的前提下，逐步把系统从“能生成草案”升级为“能越来越像熟练业务员地生成更合理草案”。
